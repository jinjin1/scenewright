# /synopsis

영상 한 편의 **정체성**을 한 페이지로 압축한다. CC 세션이 직접 LLM 추론으로 `source.txt`(책 한 챕터·아티클·인터뷰 transcript 등 ~3~5KB)를 읽고, 운영자 톤(`@VOICE.md`)에 맞춰 영상의 logline / hook / 3개 takeaway / 타겟 / 분량을 결정한다.

## 인자

`$ARGUMENTS` = `slug` (필수). `/^[a-z0-9][a-z0-9-]{0,63}$/` — `/new-episode`와 동일 규칙.

## 입력

- `episodes/<slug>/source.txt` 필수. 비어 있으면 중단하고 `/new-episode` 안내.

## 출력

`episodes/<slug>/synopsis.json` — `SynopsisSchema` (`src/schemas/synopsis.ts`) 준수.

```ts
{
  logline: string,          // 20~200자. "이 영상이 무엇을 다루는가" 한 문장
  hook: string,             // 10~200자. 0~3초에 시청자를 잡는 콜드 오픈 (VOICE.md §4 콜드 오픈: 구체 장면·반전 먼저, 인사는 그 다음)
  takeaways: [string, string, string],  // 시청자가 가져갈 3개. 각 5자 이상
  audience: string,         // 타겟 시청자 묘사. 예: "스타트업 1~3년차 PM"
  duration_min: number,     // 5~20 사이 정수. 일반적으로 10~15
  tone_notes?: string       // 이 영상 특유의 톤 메모. 생략 가능
}
```

---

## 동작

1. `$ARGUMENTS`를 `slug`로 받고 `/^[a-z0-9][a-z0-9-]{0,63}$/` 검증. 위반 시 중단.
2. `episodes/<slug>/source.txt` Read. 파일이 없거나 비어 있으면 `/new-episode <slug>`로 안내하고 중단.
3. `VOICE.md` Read (전체) + `REFERENCE/*.md` 디렉터리 목록 확인 (선택적으로 관련 카드 Read).
4. **아래 system + user prompt**를 자체 추론으로 실행해 결과 JSON 생성.
5. 결과를 `episodes/<slug>/synopsis.json`에 Write.
6. `SynopsisSchema.parse`로 검증 (또는 형상 점검). 실패하면 재생성 또는 운영자에게 보고.
7. 결과 프리뷰 출력 + `/treatment <slug>` 다음 단계 안내.

---

## System prompt

> 당신은 `@channel.config.md`에 정의된 YouTube 채널의 시니어 작가다. 채널의 주제 영역·페르소나·타겟 시청자·톤은 `@channel.config.md`와 `@VOICE.md`를 따른다.
>
> **출력 톤은 `@VOICE.md`를 반드시 준수**한다. 특히:
>
> - 후크는 **콜드 오픈**(`@VOICE.md §4`). `hook` 필드는 인사보다 **구체 장면·반전·시청자 본인 상황 한 마디를 먼저** 던져 0~3초에 잡는다("버튼 색 하나 바꿨더니 가입률이 올랐습니다. 근데 그게 진짜 버튼 때문이었을까요?"). 그 다음 비트에서 "안녕하세요. 이번 영상은 ~"으로 인사+주제. **콜드 오픈 ≠ 어그로** — 클릭베이트·통계 충격·공포 마케팅("지금 안 하면 늦습니다")·"단 하나의 비결"은 **여전히 금지**.
> - **운영자 본인 이름 등 개인 식별 정보는 영상 내레이션·hook에 넣지 말 것**. 실명 자기소개 금지 — "안녕하세요"만으로 인사 종료.
> - 영어 약어·전문 용어는 자연스러운 한글 음차로 (채널 주제에 맞는 용어; 예: 아웃컴, 디스커버리).
> - `@VOICE.md` 섹션 6 (금기 표현)을 위반하지 말 것 — "혁명적", "게임 체인저", "최고의", "정말/굉장히", "구독·좋아요" 등.
> - 시청자는 **"여러분"**. 화자는 **"저"**.
>
> **프레임워크가 등장하면 `REFERENCE/` 카드를 참조**해 정의·한계·예시의 정확성을 확보한다. `REFERENCE/`에 없는 프레임워크라면 takeaways에 포함하기 전 한 번 더 검증한다.
>
> **출력은 JSON만**. 마크다운 코드 펜스나 설명 문장 없이 순수 JSON 객체 한 개를 출력한다. 스키마는 user prompt에 명시된다.

## User prompt

> 다음 source 텍스트를 기반으로 YouTube 영상 1편의 **synopsis**를 작성하라.
>
> --- SOURCE ---
> {{source.txt 전체 내용}}
> --- END SOURCE ---
>
> 작성 규칙:
>
> 1. **logline** (20~200자): 이 영상이 무엇을 다루는가를 한 문장으로. "이 영상은 ~을 다룹니다" 같은 메타 문장 금지. 영상의 내용 자체를 압축해 한 문장으로 진술.
>    - 좋은 예: "지속적인 고객 인터뷰가 왜 분기마다의 리서치보다 PM 의사결정에 더 영향을 주는가."
>    - 나쁜 예: "이 영상에서는 고객 인터뷰의 중요성에 대해 다룹니다."
>
> 2. **hook** (10~200자): 0~3초에 시청자를 잡는 **콜드 오픈**(`@VOICE.md §4`). 인사보다 **구체 장면·반전·시청자 본인 상황 한 마디를 먼저** 던지고, "안녕하세요. 이번 영상은 ~"은 그 다음 비트로 민다. 소재 3종 중 영상 주제에 맞는 것 택1: (a) 구체 장면, (b) 반전·역질문, (c) 시청자 본인 통증. **콜드 오픈 ≠ 어그로** — 클릭베이트·통계 충격·공포 마케팅·과장어·운영자 이름 금지.
>    - 좋은 예 (구체 장면 → 반전): "버튼 색 하나 바꿨더니 가입률이 올랐습니다. 근데 그게 진짜 버튼 때문이었을까요?"
>    - 좋은 예 (시청자 본인 상황): "회의에서 제일 목소리 큰 사람 말대로 제품이 정해진 적, 다들 한 번쯤 겪어보셨을 거예요."
>    - 약한 예 (목차 낭독 — 지양): "안녕하세요. 이번 영상은 테레사 토레스의 책을 리뷰해 보려고 합니다." (인사+주제는 콜드 오픈 *다음* 비트로)
>    - 나쁜 예 (클릭베이트 — 금지): "PM의 90%가 모르는 충격적인 사실!"
>
> 3. **takeaways**: 정확히 3개. 시청자가 영상을 보고 가져갈 핵심. 각 5자 이상, 가능하면 30~80자. 추상 명사화("중요성", "방법론") 대신 행동 가능한 형태로.
>    - 좋은 예: "리서치 질문과 인터뷰 질문을 구분하는 이유와 구체 변환 방법"
>    - 나쁜 예: "고객 인터뷰의 중요성"
>
> 4. **audience**: 타겟 시청자. 직무·연차·고민 한 줄. 예: "디스커버리 루틴이 없는 스타트업 PM 1~5년차".
>
> 5. **duration_min**: 5~20 정수. source 분량과 takeaway 깊이를 보고 결정. 기본 10~15.
>
> 6. **tone_notes** (선택): 이 영상 특유의 톤 메모. 예: "책 리뷰지만 운영자 본인 실무 사례 1개 추가 권장". 일반적인 VOICE.md 톤을 다시 적지 말 것.
>
> JSON만 출력하라. 코드 펜스 없이.

---

## 검증

1. `SynopsisSchema.parse(JSON.parse(...))` 통과해야 함.
2. logline·hook·takeaways가 VOICE.md 금기 표현을 포함하지 않는지 grep로 1차 확인:
   - 위반 키워드: `"혁명적|게임 체인저|최고의|유일한|반드시.*해야|구독.*좋아요"` 등
3. takeaways 3개 중 추상 명사("중요성", "방법", "이해") 단독으로 끝나는 항목이 있으면 재생성 권장.

검증 통과 시 결과를 운영자에게 프리뷰로 보여주고 다음 단계 안내.

## 다음 단계

`/treatment <slug>` — `synopsis.json`을 받아 3개 이상의 scene 구조(`treatment.json`)로 분해.

## 비고

- LLM 호출은 CC 세션이 직접 수행. `@anthropic-ai/sdk` 사용 금지.
- 재생성이 필요하면 동일 커맨드 재실행 (덮어쓰기). source.txt를 수정한 후 재실행하면 결과가 바뀐다.
- 운영자가 결과에 동의하면 `/treatment`로 진행. 동의하지 않으면 `tone_notes`에 명시적 가이드를 추가하고 재실행하거나 source.txt를 보강.
