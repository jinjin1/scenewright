# /script-review

`/tts`(내레이션)와 `/render`(화면 텍스트)는 워딩을 **굽는다** — 어색한 표현을 그 뒤에 발견하면 재합성(~4분)·재렌더(~30분)가 든다. 이 게이트는 `/script` 직후 그 전에 워딩을 점검한다.

**하이브리드 2층 구조**:
1. **결정적 린트**(`npm run lint:script`) — 객관적으로 검출 가능한 위험(강조어 불일치·화면 영어·미음차 약어·문장끝/길이·약한 오프닝(콜드 오픈)·소스 충실도)을 자동으로 잡고 `script-review.html` 리포트를 굽는다.
2. **CC 의미 검토**(이 커맨드의 본문) — 정규식이 못 잡는 **번역체·어색한 한글·모호함**을 CC 세션이 직접 읽고 판단해 운영자 체크리스트로 정리한다.

## 인자

`$ARGUMENTS` = `slug` (필수).

## 동작

1. `slug` 검증(`/^[a-z0-9-]+$/`). `episodes/<slug>/script.json` 존재 확인 — 없으면 `/script` 먼저 안내.
2. **결정적 린트 실행**: `npm run lint:script -- <slug>`
   - 콘솔에 `error/warn/check` 요약 + 라인별 findings, `episodes/<slug>/script-review.html` 생성.
   - `source.txt`가 있으면 source-fidelity 검사도 켜진다(소스에 없는 약어/용어).
   - `treatment.json`이 있으면 **씬별 분량 결정적 점검**도 켜진다: 콘솔 `scene length:` 줄 + html "씬별 분량" 패널이 scene별 글자수%(목표 `duration_sec×9`)를 내고 90% 미달 씬을 플래그한다(단일 패스 분량 미달 ~60% 방지 — 미달 씬은 `/script`에서 그 씬만 "약 N자로 늘려라"로 재생성).
3. **script.json 전체를 읽는다**(Read). 있으면 `source.txt`도 읽어 톤·용어 기준으로 삼는다.
4. **CC 의미 검토** — 아래 루브릭으로 각 `line.text`(내레이션)와 화면 노출 텍스트(visual props의 title/heading/label/caption/body/items/highlights)를 훑는다. 결정적 findings를 출발점으로 삼되, 정규식이 못 잡은 것을 사람의 귀로 보강한다.
5. **운영자 체크리스트 출력**(아래 형식). 자동 수정하지 말고 **제안만** 한 뒤 승인 게이트.
6. 운영자가 고른 항목만 `Edit`로 `script.json`의 해당 `line.text` 또는 visual 텍스트를 수정한다. 수정 후 `npm run lint:script -- <slug>`를 한 번 더 돌려 ERROR가 0인지 확인.

## CC 의미 검토 루브릭

내레이션은 **귀로 듣는 한국어**다. "문법적으로 맞다"가 아니라 "한국어 화자가 말로 자연스럽게 하는가"가 기준.

### A. 영어 번역체 (English → Korean 직역)
- **무생물 주어 + 능동**: "이 데이터가 말해준다" → "이 데이터를 보면 알 수 있다".
- **수동태 남발**: "~에 의해 결정된다" → "~가 결정한다".
- **소유격 직역**: "우리의 목표", "당신의 고객" → 보통 한국어는 소유격 생략("목표는", "고객은").
- **"have/make/take" 직역**: "영향을 가지다"→"영향을 미치다", "결정을 만들다"→"결정을 내리다".
- **"one of the ~"**: "가장 중요한 것들 중 하나" → "가장 중요한 것" 또는 "특히 중요한".
- **접속 부사 번역체**: "그러나/하지만"의 과다, "게다가", "그러한" → 구어로.
- **명사화 과다**: "측정의 수행" → "측정하기".
- **"make/do" 직역**: "공유하게 만들어 줍니다" → "공유하게 해줍니다"; "준다" → "제공한다"(문맥).
- **명사+동사 풀어쓰기**: "논쟁을 벌이다" → "논쟁하다", "설명을 합니다" → "설명합니다".
- **"~게 되다" 남발**: "말할 수 있게 됩니다" → "말할 수 있습니다".
- **부정 "못 V"**: "못 풀고/못 정하는" → "풀지 못하고/정하지 못하는".

### A2. 어색한 한국어 — 외래어·한자어·군더더기 (실측 내부 파일럿)
> 번역문이 아니라 *한국인 PM이 실제로 말하는 입말*인지. "이 문장을 카메라 앞에서 이렇게 말할까?"
- **불필요한 외래어 → 우리말**: 덱→장표, 럭셔리→고급, 모바일 워크스테이션→이동형 업무 환경, 화이트칼라 전문직→사무직 전문가, 비즈니스 생산성→업무 생산성. (개념어 시티플로우·노스 스타·알리바이 프로그레스는 유지; 단 서술 표현 "알리바이용"→"핑계용".)
- **딱딱한 한자어 → 풀어쓰기**: 진척→일이 되는 것, 분석 마비→제대로 풀지 못하는 상황, 판정→판단, 측면→구체적으로 무엇. 관형형 한자어("예측 불가능한·일관성 없는")→서술형("들쭉날쭉하고·일관되지 않은").
- **군더더기·중복 제거**: 차근차근·진짜로·그런·뻔한·그대로 삭제; "~다는 거거든요"→"거든요"; 같은 단어 반복("거절을 자신 있게 거절"→"선택을 자신 있게").
- **문장부호·조사**: 쉼표 남발→마침표로 끊기("안녕하세요,"→"안녕하세요."), "예를 들어서,"→"예를 들어"; 영어식 복수 '들'·불필요 목적격 조사 제거("기회들에"→"기회에", "둘지를"→"둘지").

### B. 한글로 옮겨서 오히려 이상한 경우 (over-translation)
- 업계에서 **음차가 표준인 용어를 억지 한글로**: "참여도"가 어색하면 "인게이지먼트", "이탈"↔"처닝", "온보딩"을 "탑승 과정"으로 옮기는 식 금지. **운영자/PM 시청자가 실제 쓰는 말**로.
- 반대로 **영어 그대로 둬서 TTS가 못 읽을 것**(린트가 warn으로 잡음): 음차할지(예: Notion→노션) 결정.
- 직역 조어("가치 제안"이 맥락상 어색 등) — source.txt의 표현을 우선(: 소스에 없는 jargon 삽입 금지).

### C. 모호함 / TTS 위험
- **중의적 문장**: 수식 관계가 둘로 읽히는 문장 → 끊어 쓰기.
- **문장 끝 영어/핵심어**: Supertonic은 결정적이라 끝 음절이 잘리면 재합성으로 안 고쳐진다. 핵심어를 문장 끝에서 떼는 리라이트( 관련).
- **너무 긴 한 호흡**(린트 warn): 쉼표 2개 넘으면 분할 검토.

### D. 톤 (VOICE.md)
- 과장·클릭베이트("혁명적", "게임 체인저", "반드시"), 구독 구걸 — 금기.
- 운영자 본인 이름 절대 금지. 인사는 "안녕하세요"만.
- **콜드 오픈 (scene01 첫 라인 — 양면 점검)**: 영상 첫 라인이 "안녕하세요. 이번 영상은 ~"으로 시작하면 콜드 오픈이 아니다(VOICE.md §4). **내레이션**은 구체 장면·반전·시청자 본인 상황을 먼저 던지고 인사+주제는 그 다음 비트로. **화면**도 같이 가야 한다 — 첫 visual이 정적 TitleCard면 그 장면/반전을 받는 비주얼(HeroImage·StockBg·GlitchTransition·StarburstReveal·HighlightedLine)로 바꾸고 제목 카드는 인사 비트로 미룬다. 린트 `weak-opening`이 양면(내레이션 "안녕~" 시작 / 첫 visual TitleCard)을 warn으로 잡는다. **콜드 오픈 ≠ 어그로** — 강도는 올리되 클릭베이트로 가지 않는다.

### E. 이모지 (의미의 구두점 —)
- **장식 vs 의미**: `DecisionMatrix`/`ReactionBeat`의 이모지가 *판정·리스크·추세·강한 리액션*이라는 의미를 지는가? 단지 화면을 꾸미려는 장식이면 빼라(다크 에디토리얼 톤과 충돌). verdict 마커가 행 내용과 의미상 맞는지(통과면 ✅, 리스크면 ⚠️) 확인.
- **애니 이모지 1개 캡**: `ReactionBeat`이 영상당 2개 이상이면 가장 강한 1개만 남기고 나머지는 `DecisionMatrix`/정적 마커로(린트 `animated-emoji-cap`가 warn). 3D 만화 톤이라 남발 시 채널 결 훼손.
- **내레이션 금지**: `line.text`에 이모지 문자(✅🎉🔥 등)가 섞여 있으면 ERROR로 빼라 — Supertonic이 못 읽어 발음이 깨진다. 이모지는 화면 props에만.
- **스타버스트 2개 캡**: `StarburstReveal`이 영상당 3개 이상이면 가장 강한 키워드·펀치라인 1~2개만 남기고 나머지는 `HighlightedLine`으로(린트 `starburst-cap`가 warn). 레트로 광선이라 남발 시 톤이 싸진다.
- **스타버스트 = 키워드 전용**: `StarburstReveal` headline이 수치·성과(예: "3.2배", "87%")면 빼라 — hype 광선이 숫자의 권위를 깎는다(심야 홈쇼핑 느낌). 수치는 `StatHero`로(린트 `starburst-numeric-headline`가 숫자만 든 headline을 warn). 광선은 훅 반전·챕터 키워드·결론 펀치 같은 감정 1순간 전용.

## 출력 형식 (체크리스트)

우선순위 순으로, 라인별로:

```
🔴 ERROR (반드시 수정 — 안 고치면 렌더가 깨짐)
  #12 scene03-line04 · HighlightedLine
    강조어 "우리의 일"이 text에 없음 → 형광펜 안 칠해짐.
    [제안] highlights를 ["우리의 일"] → ["고객"] 으로 (text에 있는 어절).

🟡 WARN (TTS 발음/잘림 위험)
  #7 scene02-line01 · narration
    "JTBD" 음차 사전에 없음 → 발음 깨짐.
    [제안] TRANSLITERATION_MAP에 "JTBD":"제이티비디" 추가 후 /tts.

🔵 CHECK (의미 확인 — 번역체/어색함/화면 영어)
  #24 scene05-line24 · narration
    번역체: "권한을 가지고 있는" → [제안] "권한이 있는".
  #2 scene01-line01 · TitleCard.title
    화면 영어 "The Mom Test" — 책 제목이라 의도된 것으로 보임(확인만).
```

각 항목은 **[현재] → [제안 리라이트]**를 구체적으로. 확실히 의도된 것(책 제목 등)은 "확인만"으로 표시해 운영자 판단을 줄인다.

## 승인 게이트

체크리스트 끝에:
> 수정할 항목 번호를 알려주세요 (예: "12, 24 고쳐줘" / "ERROR만" / "전부"). 음차 사전 추가가 필요하면 `TRANSLITERATION_MAP`에 함께 반영합니다. 검수가 끝나면 `/storyboard`(아직 안 했으면) 또는 `/tts <slug>`로 진행하세요.

- 승인된 항목만 `Edit`로 반영. `line.text` 수정은 narration이 바뀌는 것이므로 **반드시 운영자 승인 후**.
- 음차 사전 추가 시 `tests/unit/transliterate.test.ts`에 회귀 1줄 추가(와 무관, tts 품질).
- 수정 후 `npm run lint:script -- <slug>` 재실행으로 ERROR 0 확인 → `script-review.html` 갱신.

## 다음 단계

`/storyboard <slug>`(미실행 시) → `/tts <slug>`. 워딩이 확정된 뒤 합성해야 재합성을 피한다.

## 비고

- **자동 수정 금지**: 이 게이트는 운영자 판단을 돕는 것이지 대신하는 게 아니다. 특히 번역체/어색함은 취향이 있어 제안 후 승인받는다.
- 결정적 린트만 빠르게 보고 싶으면 `npm run lint:script -- <slug>` 단독 실행 → `script-review.html`.
- LLM 호출은 이 CC 세션이 직접 수행 — SDK·API 키 사용 안 함.
