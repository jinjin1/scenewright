# VOICE.md — 대본 스타일 가이드

> **이 문서는 "대본의 톤·후크·금기 표현"을 정의합니다.**
> 음성 정체성(목소리)이 아닙니다 — 음성은 Supertonic-3 M1 voice 프리셋이 담당합니다 (`CLAUDE.md` 참조).
> 슬래시 커맨드(`/synopsis`, `/treatment`, `/script`)가 `@VOICE.md` 형태로 이 파일을 참조해서 LLM 출력 톤을 강제합니다.

> 이 문서는 한 **PM(프로덕트 매니저) 교육 채널**의 실제 톤 가이드 예시입니다. `channel.config.md`를 여러분 채널 주제로 바꾸고, 이 파일을 참고해 여러분만의 `VOICE.md`를 작성하세요.

## 1. 어조 (Tone)

PM 동료에게 책·방법론·실무 인사이트를 공유하는 톤. 진지하지만 격식이 너무 무겁지 않고, 강의·세미나에서 차근차근 풀어가는 분위기. 단정적 선언은 핵심 주장에만 한정하고, 그 외에는 "~인 거 같습니다", "~로 볼 수 있는데요" 같은 부드러운 표현으로 여지를 둠. 1인칭 사색이 아니라 1인칭 공유 — "저는 이렇게 이해했습니다, 여러분도 한번 보세요" 같은 결.

## 2. 페이스 (Pace)

문장이 긴 편이고 종속절을 여러 층 쌓아 맥락을 풀어감. 한 문장에 핵심 + 부연 + 예시까지 들어가는 경우도 잦음. 짧은 문장으로 끊어가는 리듬보다는, 한 호흡으로 한 개념을 끝까지 설명하고 잠깐 쉬어가는 리듬. TTS 합성 고려하면 너무 긴 문장은 50~70자 단위로 끊되, 인위적으로 짧게 자르지는 말 것.

## 3. 인칭과 호칭

- 화자 = **"저", "제가"** — **영상 내레이션에 운영자 본인 이름은 절대 등장 금지**. 인사는 "안녕하세요" 한 마디로 끝.
- 시청자 = **"여러분", "여러분들께서"** — 친근하지만 격식 유지
- 팀/실무 주체 = **"우리가", "우리 팀"** (포용형)
- 외부 인물·저자는 그대로 (테레사 토레스, 클레어, 레니 등)
- PM 직무 호칭은 영어 그대로 음차: "PM", "프로덕트 매니저", "프로덕트 트리오", "디자이너", "엔지니어"

## 4. 후크 패턴 (Opening Hooks) — 콜드 오픈

> ⚠️ **방향 전환 (2026-06-04, 운영자 피드백)**: 기존 가이드는 "강한 후크 안 씀 / 평이한 인사 먼저"였다. 그런데 실측상 제작된 7편 중 6편이 **"안녕하세요. 이번 영상은 ~ 살펴보려고 합니다"** 한 템플릿으로 시작해 0~15초에 시청자를 잡지 못했다(목차 낭독 ≠ 후크). 운영자가 "시작이 약하다, 후크가 필요하다"고 판단 → **콜드 오픈으로 전환**한다.

**콜드 오픈**: 인사보다 **구체 장면·반전 한 마디를 먼저** 던져 0~3초에 시청자를 잡고, **"안녕하세요. 이번 영상은 ~"은 그 다음 비트**로 민다. 콜드 오픈 소재 3종(영상 주제에 맞게 택1):

- **(a) 구체 장면**: "버튼 색 하나 바꿨더니 가입률이 올랐습니다." — 작은 구체 사실/사건으로 시작.
- **(b) 반전·역질문**: "근데 그게 진짜 버튼 때문이었을까요?" — 통념을 흔드는 질문.
- **(c) 시청자 본인 상황**: "회의에서 제일 목소리 큰 사람 말대로 제품이 정해진 적, 다들 한 번쯤 겪어보셨을 거예요." — 시청자가 바로 자기 얘기로 느끼는 통증(내부 파일럿에서 이미 통한 패턴).

그 **다음 비트**에서 인사+주제: "안녕하세요. 이번 영상은 이걸 가려내는 도구, A/B 테스트를 풀어보려고 합니다."

**콜드 오픈 ≠ 어그로.** 강도를 올리는 것이지 자극으로 가는 게 아니다. 아래는 **여전히 금지**:

- 클릭베이트("PM의 90%가 모르는 충격적 사실"), 통계 충격, 공포 마케팅("지금 안 하면 늦습니다"), "단 하나의 비결".
- §6 과장형·마케팅 용어("혁명적", "게임 체인저", "최고의", "유일한").
- 본인 이름("○○" 등 어떤 변형이든)은 콜드 오픈 포함 어디에도 금지 — 인사는 "안녕하세요"로 끝낸다.

**양면 일치 (필수)**: 콜드 오픈은 내레이션만의 일이 아니다. **화면도 같이 콜드 오픈**해야 한다 — scene01 첫 shot은 정적 타이틀 카드가 아니라 그 장면/반전을 받는 비주얼(구체 b-roll·HeroImage, 또는 GlitchTransition/StarburstReveal로 반전 키워드)이어야 하고, 제목 카드는 인사 비트로 미룬다. 첫 shot이 TitleCard면 콜드 오픈이 깨진다 (`/script` scene01 패턴 + `npm run lint:script`의 `weak-opening` 참조).

본론 안에서도 반전 진술을 계속 적극 활용:

- "그렇다면 PRD 자체가 문제일까요? 아니죠."
- "근데 진짜 문제는 ~"

## 5. 자주 쓰는 표현 (Signature Phrases)

운영자의 실제 발화에서 반복되는 전환·강조 표현:

- **전환**: "그래서 ~", "그리고 ~", "근데 ~"
- **예시 도입**: "예를 들어서 ~", "예를 들면 ~", "예가 될 것 같은데요"
- **부드러운 강조**: "~인데요", "~거든요", "~잖아요"
- **요약·정리**: "정리하면 ~", "결국 ~", "본질적으로 ~"
- **인용·참조**: "저자는 ~ 얘기를 합니다", "책에서 ~ 설명을 합니다", "~ 부분을 강조를 합니다"
- **마무리 신호**: "마지막으로 ~", "여기서 마무리하도록 하겠습니다"

영어 약어와 PM 용어는 **자연스럽게 한글 음차로** 등장:
- 아웃컴 / 아웃풋 / 디스커버리 / 인사이트 / 트랙션 / 오퍼튜니티 / 익스피리언스 맵 / 인터뷰 스냅샷 / 프로덕트 트리오

## 6. 금기 표현 (Banned)

LLM이 디폴트로 끼우기 쉬운 표현 + 운영자 톤과 맞지 않는 표현:

- **과장형 금지**: "혁명적", "게임 체인저", "패러다임 시프트", "획기적인", "압도적인"
- **마케팅 용어 금지**: "최고의", "유일한", "단 하나의", "필수적인"
- **의미 없는 부사 절제**: "정말", "굉장히", "매우" — 꼭 필요할 때만 (운영자도 가끔 씀)
- **영어 직역체 금지**: "우리는 ~할 필요가 있습니다", "이는 ~을 의미합니다"
- **공포 마케팅 금지**: "지금 안 하면 늦습니다", "모르면 손해"
- **YouTube 클리셰 금지**: "구독·좋아요 부탁드립니다" (운영자 기존 영상에 없음)
- **단정형 강요 금지**: "반드시 ~해야 합니다" — 운영자는 "~하는 것이 좋습니다" / "~할 수 있습니다" 톤
- **호객성 표현 금지**: "오늘만 특별히", "여러분께만"
- **운영자 본인 이름 금지**: 영상 내레이션·hook·script 라인 등 시청자에게 들리는 모든 텍스트에 운영자 이름 절대 사용 금지. "안녕하세요"만으로 인사 종료.
- **직역투 도발 어조 금지**: "왜 굳이 ~하시나요" 처럼 한국어에서 판단·핀잔으로 들리는 표현 회피. 영어 원문("Why bother...")을 직역하지 말고, "왜 ~하시나요" / "어떤 이유로 ~하시나요" / "왜 ~를 그렇게 하시나요" 같이 중립적 호기심 어조로.
- **학술 번역투 단어 금지**: "경험칙" → "기본 원칙", 그 외 딱딱한 한자 번역어("~성", "~화"의 과용 등)는 자연스러운 우리말로. 책 번역서 말투를 그대로 옮기지 말 것.
- **대화 인용 시 화자 관계에 맞는 높임말**: 가족·고객 대화를 재현·인용할 때 관계에 맞는 어투 사용. 예: 성인 자녀가 부모에게 말하는 인용은 존댓말("어떠세요?"), 인터뷰어→고객도 존댓말. 영어 원문이 반말 뉘앙스라도 한국어 정서에 맞게 조정.

## 6.5 자연스러운 한국어 — 번역투·외래어·한자어·군더더기 회피

> ⚠️ 실측(내부 파일럿, 2026-05-30): LLM 초안 내레이션에서 운영자가 42곳을 "한국인 시청자에게 어색하다"며 교정. 번역서 말투·불필요한 외래어·딱딱한 한자어·군더더기가 주범이었다. **내레이션은 번역문이 아니라 한국인 PM이 카메라 앞에서 실제로 말하는 입말**이어야 한다. 작성·검수 때 아래 패턴을 강제한다.

**A. 불필요한 외래어 → 쉬운 우리말** (쉬운 우리말이 있으면 우리말 우선):
- 덱(deck) → 장표 / 럭셔리 → 고급 / 탁월한 → 뛰어난 / 엔터테인먼트(기능) → 오락(기능) / 모바일 워크스테이션 → 이동형 업무 환경 / 화이트칼라 전문직 → 사무직 전문가 / 비즈니스 생산성 → 업무 생산성 / 전문직 → 전문가
- 단, **도메인 개념어(고유명사·프레임워크명)는 유지**: 시티플로우·프리미엄 트랜짓·노스 스타·알리바이 프로그레스·에버그린 베네핏. 다만 그 개념을 *서술적으로 풀 때*는 우리말로 — "알리바이용으로" → "핑계용으로". (음차 유지 대상은 §5 참조: 디스커버리·아웃컴 등 PM 표준 외래어.)

**B. 딱딱한 한자어/번역 명사 → 풀어쓰기**:
- 진척 → 일이 되는 것 / 분석 마비 → 제대로 풀지 못하는 상황 / 판정(이 빨라진다) → 빠르게 판단할 수 있다 / 측면 → 구체적으로 무엇 / 평가 → 판단
- Sino-Korean 관형형을 서술형으로: "예측 불가능한 출퇴근, 일관성 없는 승차 품질" → "출퇴근 시간이 들쭉날쭉하고, 승차 품질이 일관되지 않은"
- 직역 명사구 풀기: "물질적 소유라는 짐 없이" → "무언가를 꼭 소유하지 않고도"

**C. 번역체 구문 → 입말**:
- "만들어 줍니다"(make+do) → "해줍니다" / "논쟁을 벌이다"(명사+벌이다) → "논쟁하다" / "~할 수 있게 됩니다"(~게 되다 남발) → "~할 수 있습니다" / "준다" → "제공한다"(문맥상) / "못 박습니다"(관용구) → "정합니다"
- 부정 "못 V" → "V지 못하다": "못 풀고" → "풀지 못하고", "못 정하는" → "정하지 못하는"

**D. 군더더기·중복 제거**:
- filler 삭제: 차근차근(반복 남발)·진짜로(→진짜)·그런·뻔한·그대로
- 늘어지는 종결 다이어트: "~다는 거거든요" → "~거든요", "정해 따라간다는 뜻이거든요" → "말하거든요"
- 같은 단어 반복 회피: "거절을 자신 있게 거절" → "선택을 자신 있게"; "노라고 말하기가 어렵" → "거절하기가 어렵"

**E. 문장부호·호흡**:
- 쉼표 남발 대신 마침표로 끊어 한 호흡씩: "안녕하세요," → "안녕하세요.", "부르는데요," → "부르는데요."
- "예를 들어서," → "예를 들어" (군더더기 '서' + 쉼표 제거)

**F. 조사·복수·접속**:
- 영어식 복수 '들' 빼기: "기회들에" → "기회에"
- 불필요한 목적격 조사: "둘지를 정하는" → "둘지 정하는"; "회의 사이의 일" → "회의 사이에 일"
- 낭독 명료성 위해 축약형 풀기: "기회엔" → "기회에는"
- "오늘 영상" → "이번 영상"

**핵심 자기 점검 질문**: *"이 문장을 한국인 PM이 카메라 앞에서 실제로 이렇게 말할까?"* — 번역서를 읽는 느낌이 조금이라도 들면 다시 쓴다.

## 8. 참고 문단 (Reference Paragraph)

운영자의 실제 발화에서 톤을 가장 잘 보여주는 두 문단. ASR 자동 인식 오류를 깨끗이 정리한 버전:

### 참고 문단 A (영상 2 — AI 시대 PM의 정체성)

> LLM은 평균적입니다. ChatGPT 같은 LLM은 정보를 종합하고 추론하고, 그걸 포맷하고 명확하게 표현하는 데는 뛰어나요. 근데 LLM은 기존에 있던 데이터나 사례를 기반으로 학습되어 있기 때문에, 아직 세상에 없거나 누구도 말해 본 적이 없는 것에 대해서는 약합니다. 사람인 PM은 그렇지가 않습니다. 좋은 PM은 고객, 시장, 프로덕트 전략, UX, 내러티브 등에 대한 방대한 맥락을 가진 존재예요. 그래서 PM의 핵심 가치는, 세상에 없는 문제를 정의하는 시각과 맥락을 제공하는 힘이라고 저는 믿습니다.

→ 운영자의 **신념을 단정형으로 선언**하는 톤. 그 직전까지는 "~할 수 있습니다" / "~로 볼 수 있는데요"로 풀어가다가, 핵심 결론에서만 "저는 믿습니다"로 닫음.

### 참고 문단 B (영상 1 — 인터뷰 원칙)

> 우리는 고객의 의도나 의견이 아니라, 실제 행동이 드러나는 구체적인 경험을 물어봐야 합니다. 예를 들면 "마지막으로 이 제품을 쓰셨던 경험을 자세히 말씀해 주세요" 이런 식으로, 시간과 상황이 분명하게 드러나도록 부탁하는 질문을 해야 됩니다. 이렇게 해야 고객이 진짜로 무엇을 했는지, 그리고 어떤 맥락에서 결정을 내렸는지 정확히 알 수가 있습니다.

→ **개념 → 구체 예시 → 결과·이유** 3단 구조. 운영자가 본론 안에서 가장 자주 쓰는 패턴.
